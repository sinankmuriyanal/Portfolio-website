ETL Data Engineering with Uber Trip Data and GCP Products

Project Overview

As part of my data engineering journey, I built an end-to-end ETL pipeline using Python and SQL to process Uber trip data efficiently — from raw extraction through transformation to database loading and analytical reporting.

Data Extraction and Ingestion

Sourced Uber trip data from Kaggle (millions of ride records with timestamps, locations, fares, and trip details). Used Python and Pandas to load large CSVs, handle missing values, and standardize date-time formats for time-based analysis.

Data Analysis & Visualization

Conducted exploratory analysis via SQL and Power BI: ride demand patterns by hour/day/city, revenue trends & surge pricing analysis, customer segmentation, and heatmaps of peak Uber locations.

Machine Learning · Kaggle · Python

From Logistic Regression to Random Forests: My First ML Journey on the Titanic

My first machine learning competition on Kaggle — Titanic: Machine Learning from Disaster — turned into a crash course in data preprocessing, feature engineering, model building, and the harsh reality of model generalization.

EDA & Preprocessing

Identified missing values, filled them using median/mode, and formed the hypothesis: gender, age, family size, and passenger class strongly influence survival — which the historical context confirmed.

Feature Engineering

Extracted titles from names ("Mr", "Mrs", "Miss"), imputed missing ages based on titles, created an "is alone" feature, scaled fare and age, and encoded categorical variables — adding depth to the model's understanding.

Random Forest Breakthrough

With minimal preprocessing, Random Forest jumped my Kaggle score to 0.77511 — my personal best. Applied GridSearchCV with 5-fold cross-validation and feature importance plots to validate results.

Lesson Learned

XGBoost gave better training accuracy but lower test performance (0.746). A Voting Classifier ensemble landed around 0.765. Key insight: complex ≠ better. Simpler models with good features often generalize better.

AI-Powered Chatbot for Student Support: My DSE FAQ Project

I built the DSE FAQ Chatbot — an AI assistant that provides instant, accurate answers to 150+ student queries directly on Telegram, powered entirely on free-tier infrastructure, combining LLMs with RAG (Retrieval-Augmented Generation).

How the Chatbot Thinks

Splits queries into conversational vs. factual, maintains session memory via PostgreSQL, checks its cache first, uses LLM reasoning if needed, and falls back to FAISS semantic search over 150+ FAQs if confidence is below 70%.

Free-Tier Engineering

FastAPI handles queries · Render hosts backend and database · FAISS provides semantic search · Hugging Face Datasets store static content. Warm caching, async API calls, and token management ensure fast, reliable responses.

Technical Stack

Python + FastAPI · FAISS · PostgreSQL (Render) · OpenAI / Fireworks GPT-OSS-20B · Hugging Face Datasets · Telegram Bot API

Impact

Answers 150+ FAQs instantly on Telegram. Stores every interaction for data-driven improvement. Operates completely free — proving that resourceful engineering and smart design can overcome constraints.

JobRadar — AI-Powered Job Application Pipeline Built from Scratch

The Problem

After applying to 50+ jobs manually — search, filter, customise CV, apply, follow up, repeat — I decided to build the machine that runs the entire funnel.

What JobRadar Does

A 13-stage end-to-end AI job application pipeline that scrapes jobs from 6 platforms (LinkedIn, Indeed, Naukri, Instahyre, Cutshort, Bayt.com), scores each job using an LLM, generates tailored CVs, and manages the full lifecycle from scrape to offer.

Key Pipeline Stages

AI Scorer — Groq (Llama 3.3 70B) scores each job 0–10 on role match, skills, experience fit, domain fit, and location. Ghost Likelihood Score — Zero-latency heuristic flagging fake or stale postings. CV Engine — Groq selects and rewrites CV bullets per job description in DSE placement format. Apply Engine — LinkedIn Easy Apply via Selenium, 15 applications/day cap, human-in-the-loop confirmation. Dashboard — Real-time Next.js 14 pipeline view with SSE log streaming.

Stack

Python 3.11 · FastAPI · SQLite · Groq API · Playwright · Selenium · Node.js docx · Next.js 14 · TypeScript · Tailwind CSS

DSE Attendance Dashboard — Automating the Mundane for 60 MBA Students

The Problem

Attendance for our MBA batch was tracked across 9 separate Google Sheets — one per subject. Every time you wanted to know your overall attendance, you had to open each sheet, manually calculate sessions attended, and do the math yourself. I automated it.

What It Does

A live dashboard where any of the 60 MBA Business Analytics students at Delhi School of Economics logs in with their roll number and instantly sees subject-wise, month-wise, and overall attendance — plus how many classes they can still miss while staying above the 75% threshold.

The Automation Pipeline

Teachers update Google Sheets → Google Apps Script downloads each sheet as CSV every 2 hours → pushes CSVs to GitHub → triggers a GitHub Action → Node.js converter parses 9 different date formats and generates clean JSON → Vercel detects the push and redeploys in ~30 seconds. Zero manual intervention.

Dashboard Features

Overall attendance card with 75% marker · Attendance Marks card (out of 70 across 7 subjects) · Per-subject cards with monthly breakdown pills and date-wise drill-down · Can-miss / needs-to-attend hints per subject.

Stack

Next.js 14 · TypeScript · Tailwind CSS · Google Apps Script · GitHub Actions · Vercel · Node.js CSV converter
